# Селектор сессий — per-screen, не глобально до M4

**Дата:** 2026-05-11
**Контекст:** M2 — построены два flagship-экрана anti-cheat showcase

## Контекст

При построении `/demos/live-session` и `/demos/anti-cheat-replay` встал вопрос: где живёт выбор сессии? Past-self-инструкция для live-session явно сказала «селектор не делать, это эпоха M4 role-switcher». Но для replay-экрана без выбора инцидента экран теряет смысл — ревью одного и того же события каждый раз не нужно.

## Решение

Селектор сессий — **per-screen**, если он органичен задаче экрана. На M2:
- `/demos/live-session` — без селектора, пинован на SES-008 (единственная live-сессия в моках, ситуация естественно одна).
- `/demos/anti-cheat-replay` — chip-style селектор по 4 violation-сессиям (SES-003/004/005/006), это сердце экрана.

Глобальный селектор (роль + связанная фильтрация всех экранов) — отдельная задача M4 role-switcher, не дублируется в M2.

## Альтернативы

1. **Глобальный селектор сессий уже в M2.** Отвергнуто: лезет в задачу M4, преждевременная абстракция. Каждый экран имеет свой класс выбора (сессия / мотор / инцидент / контракт), один универсальный селектор не накроет все случаи без compromise.
2. **Без селектора в replay, фиксируем на SES-004.** Отвергнуто: экран без выбора инцидента — это `/demos/single-incident-replay`, а не anti-cheat showcase. Демонстрационная ценность падает (одна и та же история).
3. **Query-параметр `?session=` вместо UI-селектора.** Отвергнуто как единственная точка входа: не self-evident для зрителя демо. Решено реализовать `?session=&seek=` дополнительно как механизм deep-link из других экранов (fleet, incidents) — это уже намечено для M3, см. HANDOFF «Следующий шаг».

## Последствия

- Каждый M3-экран сам решает свою UX-логику выбора. Fleet — drill-down на карточку мотора. Engine-passport — переключение по моторам клиента. Incidents — фильтры по severity и kind. Black-box — список signed-block с фильтром по сессии.
- M4 role-switcher не заменяет per-screen селекторы, а накладывается сверху как фильтр области видимости. TMS-инженер видит весь парк; клиент — только свой; гонщик — только свою сессию.
- При появлении дублирующейся логики (например, два экрана с выбором сессии) — извлекать в общий компонент только при третьем повторении (rule of three), не раньше.
