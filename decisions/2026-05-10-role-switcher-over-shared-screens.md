# 2026-05-10 — Role-switcher над общими экранами вместо трёх отдельных дашбордов

## Контекст

Прежняя версия проекта (Virtual Pitwall под ArtLine) имела четыре отдельных UI-демо для разных ICP-сегментов: Team Demo (для команд-арендаторов), ArtLine Demo (для производителя), Fleet Control (цифровой паспорт), Track Management (треки). Это работало для маркетинг-сайта, где каждое демо «продавало» функциональность одному типу клиента.

В TMS Telos ICP плоский — заказчик один (TMS Motorsport), но в системе три уровня доступа: TMS-инженер (полный), клиент-арендатор (свой парк, ограниченно), гонщик (свои сессии, post-event). Возникает выбор архитектуры UI: делать три раздельных дашборда (как в Virtual Pitwall) или один экран с переключателем роли.

## Решение

**Один набор экранов, role-switcher в Navigation.** При смене роли меняется содержимое экрана: какие моторы видны, какие столбцы показываются, какие кнопки активны, есть ли доступ к доказательной части (криптоподпись, события-нарушения). URL и структура страниц одинаковы для всех ролей.

Roles реализуются через React Context, хранящий текущую роль; компоненты-экраны читают роль и применяют видимость данных. Mock-data единый, фильтрация на уровне презентации.

## Альтернативы

- **Три отдельных дашборда (`/tms/*`, `/client/*`, `/driver/*`)** — отвергнуто. Это создаёт впечатление трёх продуктов, тогда как Telos — один продукт с ролевой моделью. Дублирует код экранов. Усложняет поддержание консистентности (изменение в одном месте надо повторить в трёх). Прячет смысл переключателя — что разные люди видят разное в одних и тех же данных.
- **Один дашборд только для TMS-инженера, без клиента и гонщика** — отвергнуто. Презентация ([presentation.md:350](../presentation.md#L350)) явно упоминает «трёхуровневый доступ TMS / клиент / гонщик» как часть архитектуры облачной части. Без role-switcher прототип не покрывает существенную часть продукта.
- **Полная имитация login-flow с тремя пользователями** — отвергнуто. Это прототип UI, не функциональная демонстрация auth. Login-flow добавит работу без ценности для основной задачи (показать, как выглядят экраны).

## Последствия

- В каждом экране нужна явная разметка, что видно для какой роли. Это дисциплинирует — нельзя бездумно показывать все данные подряд, видимость становится частью дизайна.
- Демонстрация role-switcher сама по себе становится ценной фичей прототипа — наглядно показывает TMS, как одна и та же страница работает для разных пользователей.
- Mock data должна включать явные «доказательные» элементы (криптоподпись, события-нарушения), которые видны только TMS-роли. Иначе различие ролей не показать.
- Роль-switcher — экран M4 в ROADMAP, делается **после** того, как готовы основные экраны. Сначала строим в TMS-режиме (полный доступ), потом надстраиваем фильтрацию для клиента и гонщика.
