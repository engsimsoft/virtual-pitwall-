# Role context — scope for the prototype

**Дата:** 2026-05-11
**Статус:** Принято

## Контекст

M4 вводит role-switcher над общими экранами (TMS-инженер / клиент-арендатор /
гонщик) — решение из `2026-05-10-role-switcher-over-shared-screens.md`.
Прежде чем подключать каскадную фильтрацию во всех 6 dashboards, нужно
зафиксировать границы прототипа: что переключатель меняет, а что — нет.

## Решение

1. **Тип `Role = 'tms-engineer' | 'client' | 'driver'`.** Хранится в React
   Context (`<RoleProvider>` в `app/layout.tsx`), persist в `localStorage`
   через `useEffect`. Default — `tms-engineer` (полный доступ при первом
   заходе).

2. **Прикреплённые ID для не-TMS ролей.** Прототип не даёт UX-выбора
   конкретного клиента или гонщика. Для роли `client` пинится
   `CLI-01` (Карбон-Спорт), для `driver` — `DRV-01` (Артём Воронов,
   принадлежит CLI-01). Эти константы экспортируются из
   `lib/role/RoleContext` как `PINNED_CLIENT_ID` / `PINNED_DRIVER_ID` и
   используются дашбордами как scope для фильтрации.

3. **Переключатель — компактные pills в Navigation (landing) и в
   `DashboardTopBar` (все 6 dashboards).** Один компонент `<RoleSwitcher>`,
   ре-юзается в обоих местах. На landing Navigation структурно
   перекомпонован: logo слева, single nav-link по центру, switcher справа.

4. **Фильтрация — на клиенте, в каждом Dashboard.** Server-page.tsx по-
   прежнему собирает все данные из mockData и передаёт целиком; каскадная
   фильтрация применяется в client-component через `useRole()`. Это
   избегает протекания клиентского контекста в server-component и
   укладывается в принцип «прототип = визуализация, не data pipeline».

## Граница: что НЕ делает переключатель

- **Не аутентификация.** Это не login flow, не AuthZ. Прототип демонстрирует
  «как пользователь с такой ролью видит систему», а не блокирует доступ.
- **Не выбор конкретного субъекта.** UX выбора «какой именно клиент» или
  «какой именно гонщик» отсутствует. Один pinned ID на роль. Расширение —
  отдельный scope, не часть M4.
- **Не меняет mock-data.** Все ENGINES/SESSIONS/INCIDENTS остаются в памяти
  как есть. Фильтрация урезает только то, что видит дашборд.
- **Не SSR-консистентность.** На сервере роль всегда `tms-engineer` (default),
  на клиенте hydrate из localStorage. Первый paint может моргнуть между
  ролями для не-TMS — приемлемо для прототипа.

## Почему так

Полный role-management (выбор конкретного клиента, отдельные UI-flows для
каждой роли, server-side filter, auth) — это уже не UI-прототип, а
полу-реальное приложение. Прототип должен показывать **различие в видимой
картине** между тремя уровнями доступа, не имитировать процесс. Pinned IDs
дают это различие на наглядном примере (CLI-01 видит только свои 3-4
мотора, DRV-01 — только свои сессии) без UX-разрастания.

## Зависимые экраны

- `/demos/fleet` — фильтр списка моторов по `engine.clientId` (client) или
  по наличию сессий с `driverId` (driver).
- `/demos/engine-passport` — рестрикция `EngineSelector` к доступным моторам,
  empty state при пустом скоупе.
- `/demos/incidents` — фильтр инцидентов по engine ownership / session
  ownership.
- `/demos/live-session` — пин на SES-008 остаётся; если активная роль не
  имеет доступа (driver/client не из CLI-01) — empty state «у вас нет
  активной live-сессии».
- `/demos/anti-cheat-replay` — фильтр bundles из 4 violation-сессий.
- `/demos/black-box` — фильтр bundles, empty state при отсутствии.

Этот ADR покрывает **инфраструктурный** коммит (context + switcher без
фильтрации). Сама каскадная фильтрация — следующий коммит, с возможным
follow-up ADR если возникнут не-тривиальные решения (например, что
показывать на landing для роли с пустым скоупом).
